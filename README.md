SA Hotel Appium Java TestNG Framework

A production-grade Java Appium + TestNG mobile automation framework built on a real hotel booking app. This project focuses on reliability, parallel execution, Page Object Factory, CSV-driven data, and CI-first reporting with Allure history and JUnit XML for Jenkins.

If you want an automation suite that behaves like real engineering software and not a demo, you are in the right place.

✨ Speciality

Real-world scope for a travel and hotel booking app with end-to-end flows.

Stable locator strategy for apps without accessibility ids, using constrained XPath and targeted UiSelector.

Parallel independence with dynamic port allocation and per-thread drivers.

Explicit test state with a @ResetPolicy annotation to control reset behavior per test.

CI-first reporting that preserves Allure history and produces JUnit XML for Jenkins.

Device agility through testng.xml so you can swap emulators or devices without code changes.

📌 Project Description

I built this framework after weeks of iteration to solve real testing problems: flaky locators, overscrolls, session resets, and brittle CI. The early devices, including a Google tablet and a Pixel 2 emulator, did not expose accessibility ids, so I hardened the locator strategy. I constrained XPath to short, anchored expressions, used UiSelector where text or id was stable, and replaced blind sleeps with intent-based waits like presence, visibility, and clickability.

I moved to a stable duo of Pixel 2 and Pixel 9 Pro XL and kept the device matrix flexible through testng.xml. Dynamic system ports and clean per-thread lifecycles removed collisions. Page Object Factory keeps locators centralized and lazily resolved. CSV-driven data providers scale coverage without code edits. On any failure, the framework captures a screenshot, a short video clip, device logs, and a JUnit report so humans and CI see the same truth. Allure history is preserved between runs to make trends actionable.

🚀 Key Features

Covered journeys: login and logout, hotel search with filters, profile update, and hotel booking with positive and negative validations.

Architecture: Page Object Factory with @AndroidFindBy and AppiumFieldDecorator for lazy, resilient elements.

Data: CSV-driven data providers for signup, search, and booking scenarios.

Reset strategy: per-test @ResetPolicy for NO_RESET or RESET_DATA, with additional modes for fast and full resets.

Parallel execution: device-level independence, dynamic ports, isolated artifacts, and no shared mutable state.

Artifacts on failure: screenshots, short video clips, device or console logs, and JUnit XML for CI.

Reporting: Allure with steps, parameters, and attachments, plus preserved history for real trend analysis.

CI integration: Jenkins pipeline friendly. Archives artifacts and publishes Allure. Pass or fail is gated by JUnit XML.

Run flexibility: run single tests or full suites from the IDE or Maven. Works for quick local debugging and large CI fans.

▶️ How to Run
Run a single test locally from Maven
mvn -Dtest=tests.Test4 test

Run the full suite with testng.xml
mvn -DsuiteXmlFile=testng.xml test

Run in parallel

Use testng.xml to set parallel methods or classes, and define device parameters per <test> block. The framework allocates ports dynamically so multiple sessions do not collide.

Example testng.xml device parameters:

<!DOCTYPE suite SYSTEM "https://testng.org/testng-1.0.dtd" >
<suite name="Android Suite" parallel="tests" thread-count="2">
  <test name="Pixel_2_Run">
    <parameter name="deviceName" value="Pixel_2"/>
    <parameter name="udid" value="emulator-5554"/>
    <parameter name="systemPort" value="8201"/>
    <classes>
      <class name="tests.Test1"/>
      <class name="tests.Test4"/>
    </classes>
  </test>

  <test name="Pixel_9_Pro_XL_Run">
    <parameter name="deviceName" value="Pixel_9_Pro_XL"/>
    <parameter name="udid" value="emulator-5556"/>
    <parameter name="systemPort" value="8203"/>
    <classes>
      <class name="tests.Test1"/>
      <class name="tests.Test4"/>
    </classes>
  </test>
</suite>


You can swap device names and UDIDs here without changing code. The framework keeps sessions and artifacts isolated.

🧩 Page Object Factory Example
public class HomePage {
    private final AndroidDriver driver;
    private final WebDriverWait wait;

    @AndroidFindBy(xpath = "//android.widget.FrameLayout[@content-desc='More']")
    private WebElement moreButton;

    @AndroidFindBy(id = "com.myhotels.sa:id/citySearchLayout")
    private WebElement citySearchLayout;

    @AndroidFindBy(uiAutomator = "new UiSelector().text(\"Start Search\")")
    private WebElement startSearchButton;

    public HomePage(AndroidDriver driver) {
        this.driver = driver;
        PageFactory.initElements(new AppiumFieldDecorator(driver, Duration.ofSeconds(10)), this);
        this.wait = new WebDriverWait(driver, Duration.ofSeconds(15));
    }

    public void tapMore() {
        wait.until(ExpectedConditions.elementToBeClickable(moreButton)).click();
    }
}

🧰 Jenkins CI Tips

Use Maven goals clean test.

Archive allure-results, allure-report, junit-reports, and any media directories for screenshots and videos.

Publish Allure with history retained between builds.

Gating is based on JUnit XML so failing tests reflect as build failures.

🩺 Troubleshooting

No accessibility ids: prefer short, anchored XPath or stable UiSelector with clear text or id. Avoid long chains.

Parallel flakiness: verify unique systemPort, use per-thread drivers, and keep artifacts namespaced.

Overscroll or missed elements: target the correct container with UiScrollable and stop as soon as the target appears.

State leakage: set @ResetPolicy per test so sensitive flows start clean while others stay fast.

Allure history not showing: ensure history/ is copied into allure-results/ before report generation.
